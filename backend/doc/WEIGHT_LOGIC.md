# 权重计算逻辑

本文档说明 LetouMe 项目中的两套权重体系：

1. AI 预测阶段的选号评分权重
2. 前端展示阶段的模型历史评分与号码汇总加权

这两套逻辑用途不同，不应混淆。

## 一、AI 预测阶段的权重逻辑

这部分来源于 `backend/doc/dlt_prompt2.0.md`，用于约束模型生成 5 组预测方案。

### 1. 热号策略：多周期加权频率

前区每个号码的热度得分：

```text
热度得分 =
  近5期出现次数 × 5 +
  近10期出现次数 × 3 +
  近30期出现次数 × 2
```

附加修正：
- 如果该号码在上一期出现：得分 × 0.6
- 如果该号码连续 4 期未出现：得分 × 0.8

后区热号策略：
- 统计最近 20 期后区号码频率
- 优先选择近 20 期最活跃的号码

### 2. 冷号策略：遗漏加权

前区每个号码的遗漏得分：

```text
如果遗漏期数在 4-8 期：遗漏期数 × 1.0
如果遗漏期数在 9-16 期：遗漏期数 × 1.4
其他情况：遗漏期数 × 0.8
```

回温加成：

```text
如果某号码遗漏超过 10 期，
且在最近 4 期内首次回补：得分 × 1.3
```

后区冷号策略：
- 优先选择遗漏 4-10 期的后区号码
- 避免全部选择刚开出不久的号码

### 3. 趋势策略：多周期频率差

前区趋势分公式：

```text
趋势分 =
  (5期频率/5 - 30期频率/30) × 100 +
  (10期频率/10 - 30期频率/30) × 50
```

解释：
- 趋势分大于 0：视为上升趋势
- 趋势分小于 0：视为下降趋势

周期回补加成：

```text
如果某号码连续 3 期未出现，
但最近 2 期出现回补：趋势分 +15
```

后区趋势/节奏逻辑：
- 计算每个后区号码的历史平均遗漏期
- 优先选择当前遗漏期数接近历史平均遗漏期的号码

### 4. 综合策略：多维评分融合

前区综合得分：

```text
综合得分 =
  热度得分 × 0.30 +
  遗漏得分 × 0.25 +
  平衡得分 × 0.20 +
  周期得分 × 0.25
```

后区综合得分：

```text
后区综合得分 =
  热度得分 × 0.35 +
  遗漏得分 × 0.30 +
  周期得分 × 0.20 +
  中频得分 × 0.15
```

说明：
- 这些公式是 Prompt 中规定的量化框架
- 模型需要在这些规则约束下输出 5 组不同策略方案
- Python 后端本身不逐项重算这些综合分，而是负责组织历史数据、调用模型、校验输出结构

## 二、前端展示阶段的权重逻辑

这部分来源于 React 前端中的历史评分与号码汇总逻辑，用于模型评分和号码汇总展示，不参与实际选号生成。

### 1. 模型历史评分窗口

前端固定只看最近：

```text
20 期
```

对应常量：

```text
SPORT_LOTTERY_SCORE_WINDOW = 20
```

### 2. 单组预测命中分

对某个模型在某一期中的某一组预测，先计算：

```text
redScore = 红球命中数 / 5
blueScore = 蓝球命中数 / 2
groupScore = (redScore + blueScore) / 2
```

### 3. 单期模型分

对某模型在某一期的全部预测组：

```text
bestScore = 该期所有组得分中的最大值
avgScore = 该期所有组得分的平均值
periodScore = bestScore × 0.6 + avgScore × 0.4
```

对应常量：

```text
SPORT_LOTTERY_BEST_SCORE_WEIGHT = 0.6
SPORT_LOTTERY_AVG_SCORE_WEIGHT = 0.4
```

### 4. 模型历史总评分

对最近 20 期的 `periodScore` 取平均，再乘 100 并四舍五入：

```text
score100 = round(最近窗口内 periodScore 平均值 × 100)
```

前端同时展示：
- `bestComponent`：最近窗口内 `bestScore` 的平均值 × 100
- `avgComponent`：最近窗口内 `avgScore` 的平均值 × 100

最终每个模型会得到：
- `score100`
- `bestComponent`
- `avgComponent`
- `sampleSize`
- `windowSize`

### 5. 模型排序权重

预测卡片的模型排序逻辑：

1. 用户手动置顶的模型优先
2. 其余模型按 `score100` 从高到低排序
3. 分数相同则按原始顺序兜底

## 三、前端号码汇总的加权逻辑

前端在“模型预测汇总”区域支持两种模式：

1. 不加权统计
2. 按历史评分加权

### 1. 不加权模式

默认情况下：
- 每个模型的每一组预测，某号码出现一次就记 `1`
- 号码出现次数就是原始累计出现次数

### 2. 按历史评分加权模式

当用户打开“按历史评分加权”后，每个模型的权重为：

```text
scoreWeight = score100 / 100
```

例如：
- 模型历史评分 `82`，则该模型的预测权重为 `0.82`
- 模型历史评分 `55`，则该模型的预测权重为 `0.55`

此时：
- 每个号码每出现一次，不再加 `1`
- 而是加对应模型的 `scoreWeight`

某号码的加权出现次数：

```text
weightedCount = 所有命中该号码的预测组权重之和
```

总预测量也同步变为加权总预测量：

```text
weightedTotalPredictions += model.predictions.length × scoreWeight
```

最终展示的命中率：

```text
rate = 号码出现次数 / 总预测量
```

其中：
- 不加权模式：分子分母都用原始次数
- 加权模式：分子分母都用加权值

这套逻辑只影响前端汇总展示，不会改写数据库中的原始预测记录。

## 说明

- 本项目中的权重和评分是启发式量化规则，不代表统计显著性证明
- AI 预测结果仅供参考，不构成任何投注建议
- 前端历史评分只反映最近窗口内的命中表现，不代表长期稳定性
- Prompt 中的权重用于约束模型生成，不等同于后端存在一套完全确定性的公式引擎
